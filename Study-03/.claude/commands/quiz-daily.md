매일 퀴즈 데이터를 점검하고, 부족한 카테고리에 새 문제를 추가하는 일일 관리 작업을 수행합니다.

대상 디렉터리: `quiz-game/src/data/`
대상 파일: `bible.ts`, `hiphop.ts`, `nba.ts`, `mlb.ts`

**중요: 각 단계는 순서대로 실행하며, 어떤 단계에서든 검증 실패 시 즉시 중단하고 오류를 보고합니다.**

---

## STEP 1: 파일 구조 읽기 및 이해

`quiz-game/src/data/` 디렉터리의 4개 파일(bible.ts, hiphop.ts, nba.ts, mlb.ts)을 모두 읽습니다.

검증 항목:
- [ ] 4개 파일이 모두 존재하는가?
- [ ] 각 파일이 `Question[]` 타입 배열을 default export 하는가?
- [ ] 각 문제 객체가 `id`, `category`, `question`, `options`, `answer`, `explanation` 필드를 모두 포함하는가?

실패 조건:
- 파일이 누락되면 → `[STEP 1 실패] {파일명} 파일이 존재하지 않습니다.` 출력 후 중단
- 필수 필드가 누락된 문제가 있으면 → `[STEP 1 실패] {문제ID}에 {필드명} 필드가 누락되었습니다.` 출력 후 중단

성공 시 출력:
```
[STEP 1 완료] 4개 파일 확인, 총 N개 문제 로드
```

## STEP 2: 현재 문제 개수와 분포 확인

각 카테고리별로 아래 정보를 집계합니다.

수집 항목:
- 카테고리별 문제 수
- 카테고리별 정답 분포 (answer 값 0, 1, 2, 3 각각의 개수)
- 카테고리별 난이도 분포 (상/중/하 — 아래 기준으로 판정)
  - 상: 구체적 수치·통계·연도 등 암기 없이는 풀 수 없는 문제
  - 중: 해당 분야 관심자가 추론 가능하지만 정확한 지식 필요
  - 하: 일반 상식 수준

검증 항목:
- [ ] 모든 문제의 `answer` 값이 0~3 범위인가?
- [ ] 모든 문제의 `options` 배열 길이가 정확히 4인가?
- [ ] 모든 문제의 `id` 형식이 `{category}_{3자리 숫자}`인가?
- [ ] 각 카테고리 내 id 번호가 연속적인가? (빠진 번호 없는지)

실패 조건:
- `answer` 값이 범위 밖이면 → `[STEP 2 실패] {문제ID}의 answer 값이 유효 범위(0~3)를 벗어남: {값}` 출력 후 중단
- `options` 길이가 4가 아니면 → `[STEP 2 실패] {문제ID}의 options 개수가 4가 아님: {개수}개` 출력 후 중단
- id 형식이 틀리면 → `[STEP 2 실패] {문제ID}의 id 형식이 규칙에 맞지 않음` 출력 후 중단

성공 시 출력:
```
[STEP 2 완료] 분포 분석 완료
| 카테고리 | 문제 수 | 0번 | 1번 | 2번 | 3번 | 난이도(상/중/하) |
|----------|---------|-----|-----|-----|-----|-----------------|
| bible    | N개     | N   | N   | N   | N   | N/N/N           |
| hiphop   | N개     | N   | N   | N   | N   | N/N/N           |
| nba      | N개     | N   | N   | N   | N   | N/N/N           |
| mlb      | N개     | N   | N   | N   | N   | N/N/N           |
```

## STEP 3: 카테고리별 부족한 부분 파악

STEP 2의 결과를 기반으로 각 카테고리의 개선 필요 사항을 분석합니다.

분석 기준:
1. **정답 편중**: 특정 answer 번호가 해당 카테고리 문제의 40% 초과 시 경고
2. **정답 공백**: 특정 answer 번호가 0개인 경우 경고
3. **난이도 편중**: '하' 난이도가 존재하면 마니아 퀴즈 취지에 맞지 않으므로 경고
4. **주제 다양성**:
   - bible: 구약/신약 비율 확인 (70% 초과 편중 시 경고)
   - hiphop: 특정 연대(80s/90s/2000s/2010s)가 60% 초과 시 경고
   - nba: 동일 선수 3회 이상 출제 시 경고
   - mlb: 동일 선수 3회 이상 출제 시 경고
5. **문제 수 불균형**: 카테고리 간 문제 수 차이가 3개 이상이면 경고

분석 결과를 아래 형식으로 정리합니다:
```
[STEP 3 완료] 개선 필요 사항 N건 발견

카테고리별 개선 사항:
- {카테고리}: {구체적 내용} (우선순위: 높음/보통/낮음)
```

이 분석 결과는 STEP 4에서 새 문제 추가 시 우선순위를 결정하는 데 사용합니다.

---

## STEP 4: 새 문제 추가 전 중복 체크 및 추가

STEP 3에서 파악한 부족한 부분을 기반으로, **가장 개선이 필요한 카테고리 1개**를 선택하여 문제 1개를 추가합니다.

카테고리 선택 우선순위:
1. 문제 수가 가장 적은 카테고리
2. 동일 문제 수라면 정답 편중이 가장 심한 카테고리
3. 그래도 동일하면 bible > hiphop > nba > mlb 순

새 문제 작성 전 중복 체크:
- 해당 카테고리의 모든 기존 문제 `question` 텍스트를 확인
- 같은 인물/앨범/팀/사건을 다루는 문제가 이미 있는지 확인
- 중복 발견 시 → 다른 주제로 문제를 작성

중복 체크 결과 출력:
```
[STEP 4-1] 중복 체크 완료 — 선택 카테고리: {카테고리}
- 기존 문제에서 다룬 주제: {주제1}, {주제2}, ...
- 새 문제 주제: {선택한 주제} (중복 없음 확인)
```

새 문제 작성 규칙:
- id: 마지막 번호 + 1 (`{category}_{3자리}` 형식)
- answer: 해당 카테고리에서 가장 적게 사용된 번호 배정 (동률 시 0 > 3 > 1 > 2 우선)
- 난이도: STEP 3에서 부족한 난이도로 작성 (기본값: 상)
- CLAUDE.md의 카테고리별 출제 방향 준수
- CLAUDE.md의 검증 가이드라인 6개 항목 모두 충족

검증 체크리스트 (모두 통과해야 추가):
- [ ] 정답 유일성
- [ ] 최상급 표현 기준 명시
- [ ] 시간·범위 명확성
- [ ] 교차 검증 (WebSearch로 사실 확인)
- [ ] 기존 문제 중복 없음
- [ ] 오답 선택지 그럴듯함

실패 조건:
- 검증 체크리스트 중 하나라도 불통과 시 → `[STEP 4 실패] 검증 항목 '{항목명}' 불통과: {사유}` 출력 후 중단
- 파일 쓰기 실패 시 → `[STEP 4 실패] {파일명} 쓰기 실패` 출력 후 중단

성공 시 출력:
```
[STEP 4-2] 문제 추가 완료
- 카테고리: {카테고리}
- 문제 ID: {id}
- 문제: {question 텍스트}
- 정답: {answer번호}번 — {정답 텍스트}
- 검증: 6/6 통과
```

## STEP 5: 추가 후 형식 검증

문제가 추가된 파일을 다시 읽어서 아래 항목을 검증합니다.

검증 항목:
- [ ] TypeScript 구문 오류 없음 (`npm run build` 또는 파일 파싱으로 확인)
- [ ] 새로 추가된 문제의 모든 필드가 올바른 타입인가?
  - `id`: string, `{category}_{3자리}` 형식
  - `category`: 유효한 Category 타입값
  - `question`: 비어있지 않은 string
  - `options`: 정확히 4개의 string 배열
  - `answer`: 0~3 범위의 number
  - `explanation`: 비어있지 않은 string
- [ ] 배열 내 쉼표, 괄호 등 구문이 올바른가?
- [ ] 기존 문제가 변경되지 않았는가? (기존 문제 수 유지 확인)

실패 조건:
- 구문 오류 → `[STEP 5 실패] {파일명}에 구문 오류 발견: {오류 내용}` 출력 후 중단
- 필드 타입 오류 → `[STEP 5 실패] {문제ID}의 {필드명} 타입이 올바르지 않음` 출력 후 중단
- 기존 문제 수 변동 → `[STEP 5 실패] 기존 문제가 변경됨 (이전: N개, 현재: N개)` 출력 후 중단

성공 시 출력:
```
[STEP 5 완료] 형식 검증 통과
- 파일: {파일명}
- 총 문제 수: N개 (기존 N + 신규 1)
- 구문 오류: 없음
- 타입 검증: 통과
```

## STEP 6: 전체 데이터 백업

4개 데이터 파일을 `quiz-game/src/data/backup/` 디렉터리에 백업합니다.

백업 규칙:
- 백업 디렉터리가 없으면 생성
- 파일명 형식: `{원본파일명}.{YYYYMMDD}.bak.ts`
  - 예: `bible.20260617.bak.ts`
- 같은 날짜의 백업이 이미 있으면 덮어쓰기
- 4개 파일 모두 백업

실패 조건:
- 디렉터리 생성 실패 → `[STEP 6 실패] backup 디렉터리 생성 실패` 출력 후 중단
- 파일 복사 실패 → `[STEP 6 실패] {파일명} 백업 실패` 출력 후 중단

성공 시 출력:
```
[STEP 6 완료] 백업 완료
- 백업 경로: quiz-game/src/data/backup/
- 백업 파일: {파일1}, {파일2}, {파일3}, {파일4}
- 백업 일시: {YYYY-MM-DD HH:MM}
```

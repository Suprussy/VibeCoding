# PRD — Step 2: 레시피 생성

## 1. 개요
Step 1에서 얻은 재료 목록과 사용자가 입력한 옵션을 바탕으로, 텍스트 LLM이 실행 가능한 레시피를 생성한다. 본 단계의 출력물은 Step 3에서 저장 대상이 된다.

## 2. 목표
- Step 1의 재료 배열을 입력으로 받아 1~3개의 레시피 후보를 생성한다.
- OpenRouter를 통해 호출하는 모델은 **`nvidia/nemotron-3-super-120b-a12b:free`** 로 고정한다.
- 결과는 화면 표시뿐 아니라 Step 3에서 그대로 저장 가능한 구조화된 JSON이어야 한다.

## 3. 의존성
- Step 1이 반환하는 재료 배열(`ingredients[]`)이 입력으로 사용된다.
- `OPENROUTER_API_KEY` 환경변수는 Step 1과 공유한다.

## 4. 사용자 스토리
- (US-1) 사용자로서 인식된 재료로 만들 수 있는 레시피를 추천받고 싶다.
- (US-2) 사용자로서 알레르기/식이제한, 인분 수, 조리 시간 상한, 난이도, 요리 카테고리(한/중/일/양식 등)를 지정하고 싶다.
- (US-3) 사용자로서 추천 결과가 마음에 들지 않으면 "다시 추천" 버튼으로 새 후보를 받고 싶다.
- (US-4) 사용자로서 부족한 재료가 있다면 "추가 구매 필요 재료"로 명시적으로 보고 싶다.

## 5. 기능 요구사항
### 5.1 입력 패널
- Step 1 결과로부터 자동 채워진 재료 칩(편집 가능).
- 사용자 옵션:
  - 인분: 1 ~ 6 (기본 2)
  - 최대 조리 시간(분): 슬라이더 10 ~ 120 (기본 30)
  - 난이도: easy / medium / hard
  - 알레르기/제외 재료: 자유 텍스트 + 칩
  - 식이 옵션: 채식, 비건, 글루텐프리 등 다중 선택
  - 카테고리 선호: 한식/일식/중식/양식/무관

### 5.2 서버 처리
- 라우트: `POST /api/recipes`
- 페이로드:
  ```json
  {
    "ingredients": [{"name": "토마토", "quantity_estimate": "3개"}],
    "preferences": {
      "servings": 2,
      "max_minutes": 30,
      "difficulty": "easy",
      "exclude": ["땅콩"],
      "diet": ["vegetarian"],
      "cuisine": "한식"
    }
  }
  ```
- 서버는 OpenRouter Chat Completions에 다음을 호출:
  - 모델: `nvidia/nemotron-3-super-120b-a12b:free`
  - `response_format: { type: "json_object" }` (모델이 지원하는 경우) 또는 시스템 프롬프트로 JSON-only 강제.
  - `temperature` 기본 0.7, "다시 추천"시 0.95로 상향.

### 5.3 프롬프트 설계
- 시스템 메시지:
  - 역할: "당신은 한국어로 답하는 요리 도우미다."
  - 제약: 사용자 재료를 최대한 활용하고, 부족한 재료는 별도 필드(`missing_ingredients`)로 분리.
  - 안전: 알레르기/제외 재료는 절대 포함하지 않는다.
  - 출력: JSON만, 마크다운/코드펜스 금지.
- 사용자 메시지: 5.2 페이로드를 그대로 직렬화하여 전달.

### 5.4 출력 스키마
```json
{
  "recipes": [
    {
      "id": "uuid-v4",
      "title": "토마토 계란 볶음",
      "cuisine": "한식",
      "difficulty": "easy",
      "estimated_minutes": 15,
      "servings": 2,
      "used_ingredients": [
        {"name": "토마토", "amount": "2개"}
      ],
      "missing_ingredients": [
        {"name": "간장", "amount": "1큰술"}
      ],
      "steps": [
        {"order": 1, "instruction": "토마토를 6등분으로 자른다."}
      ],
      "tips": ["계란을 먼저 반숙으로 익혀두면 부드럽다."],
      "tags": ["빠른요리", "1인분가능"]
    }
  ]
}
```
- `recipes.length` 는 1 ~ 3.
- `id` 는 서버에서 생성(클라이언트 신뢰 금지).
- 서버는 응답을 검증하고 스키마 위반 시 1회 재시도, 실패 시 5.6 에러 처리.

### 5.5 결과 화면
- 레시피 카드 리스트(제목, 소요시간, 난이도, 인분, 태그).
- 카드 클릭 시 상세 패널: 사용 재료, 부족 재료(쇼핑 리스트로 복사 가능), 단계별 조리법, 팁.
- 액션:
  - "다시 추천": 동일 입력으로 재호출(temperature 상향).
  - "옵션 수정": 5.1 패널로 돌아가기.
  - "이 레시피 저장" → Step 3으로 연결(Step 3 미구현 시 비활성).

### 5.6 에러 처리
| 케이스 | 처리 |
| --- | --- |
| 재료 배열이 비어 있음 | 호출 차단, "재료를 1개 이상 입력하세요" 표시 |
| OpenRouter 4xx/5xx | 재시도 1회 후 사용자에게 재시도 버튼 노출 |
| 타임아웃(>45s) | 자동 취소, 재시도 안내 |
| JSON 파싱/스키마 실패 | 1회 재시도, 실패 시 "레시피 생성에 실패했습니다" |
| 알레르기 재료가 결과에 포함됨(서버 검증) | 결과를 폐기하고 1회 재시도, 그래도 실패하면 사용자에게 경고 |

## 6. UI/UX
- 페이지 라우트: `/recipes` (Step 1 결과를 쿼리/스토어로 수신).
- 카드 그리드는 모바일 1열 / 데스크톱 2~3열.
- 생성 중에는 스켈레톤 카드 + 진행률 텍스트.
- 단계별 조리법은 체크리스트로 표시(완료 표시는 세션 한정, 영속화는 Step 3 이후).

## 7. API 요약
- `POST /api/recipes` → `{ recipes: Recipe[] }`
- `POST /api/recipes/regenerate` (선택) → 동일 페이로드 + `seed`로 다양성 확보.

## 8. 완료 조건 (Acceptance Criteria)
- [ ] 재료 배열과 사용자 옵션을 받아 1~3개의 레시피를 생성한다.
- [ ] 응답 JSON이 5.4 스키마를 100% 만족한다(서버 검증).
- [ ] 알레르기/제외 재료가 결과에 절대 포함되지 않는다(서버 검증 + 회귀 테스트).
- [ ] "다시 추천" 시 이전 결과와 다른 후보가 1개 이상 포함된다(직전 응답 ID/제목 비교).
- [ ] 평균 응답 시간 ≤ 45초.

## 9. 비범위 (Out of Scope)
- 영양 정보(칼로리/탄단지) 자동 계산 — 향후 별도 단계.
- 레시피 영상/이미지 생성.
- 다국어 출력(현 단계는 한국어 고정).
- 로그인 사용자 기반 개인화(Step 3에서 다룸).
